What is fuzzy matching?

A non-technical walk through where fuzzy matching fits in your work, when to reach for it, and the three explicit stages of how it actually figures out a match.

Fuzzy matching is how a computer figures out that two strings probably mean the same thing, even when they aren't identical. "Jhon Smith" almost certainly meant "John Smith". A human reads that instantly. A computer needs a small pipeline.

N°1 Where it sits

Fuzzy matching lives inside data cleansing

When an analyst is cleaning data, they're really doing three jobs at once. Validation checks values are sensible. Standardisation forces a consistent format. Record linkage matches messy records to a known list. Fuzzy matching is one technique inside that last job — the one that handles imperfection. 

DATA CLEANSING  (making messy data usable)
│
├── VALIDATION
│   Is this even a valid value?
│       • Does this look like a properly-formatted email?
│       • Is this postcode 4 digits?
│       • Is this date plausible (not the year 9999)?
│       • Is this name field empty when it shouldn't be?
│
├── STANDARDISATION
│   Force every record into the same format.
│       • Phone:    "0400 000 000"   ──►  "+61 400 000 000"
│       • Address:  "10 Smith St"    ──►  "10 Smith Street"
│       • Name:     "JOHN SMITH"     ──►  "John Smith"
│       • Date:     "1/2/24"         ──►  "2024-02-01"
│
└── RECORD LINKAGE
    Match a record to a known entity in your reference list.
    │
    ├── EXACT MATCH  — strings must be identical.
    │                  First pass — cheap and unambiguous.
    │
    └── FUZZY MATCH  — tolerate typos and differences.
                       Second pass — when exact match fails.
                       ◄── this tool

Standardisation vs normalisation — they overlap in technique but live at different levels. Standardisation is the dataset-level practice ("every phone number in our system looks the same"). Normalisation is the per-string step inside fuzzy matching, applied to a single value just before it gets compared. You'll see normalisation in the pipeline below.

N°2 When to reach for it

Exact first, fuzzy second

Exact match and fuzzy match aren't alternatives — they run as a cascade. Exact match is cheap and certain, so you do it first. Fuzzy match is the expensive backup that handles the long tail of typos, abbreviations, and reordering. 

For each messy record:

   1. EXACT MATCH against the reference list
        │
        ├─ Found     ──►  Done. Match resolved.
        │
        └─ Not found
                │
                ▼
   2. FUZZY MATCH  (returns scored candidates — this tool)
                │
                ▼
   3. Apply a confidence threshold
        │
        ├─ HIGH score    ──►  Auto-match
        ├─ MEDIUM score  ──►  Human review queue
        └─ LOW score     ──►  Leave unmatched, flag for follow-up
N°3 How it works, step by step

The three stages of a fuzzy match

Inside the box marked fuzzy match, three explicit stages run in order. The same worked example flows through all three so you can see the data transform. 

┌──────────────────────── FUZZY MATCHING ─────────────────────────┐
│                                                                 │
│  YOU TYPE                                                       │
│  "Jhon Smith"                                                   │
│      │                                                          │
│      ▼                                                          │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  STAGE 1 · NORMALISATION  (a.k.a. cleaning)             │    │
│  │  Tidy the string into a standard form. Rule-based,      │    │
│  │  deterministic — same input always gives same output.   │    │
│  │                                                         │    │
│  │      "Jhon Smith"   ──►   "jhon smith"                  │    │
│  └─────────────────────────────────────────────────────────┘    │
│      │                                                          │
│      ▼                                                          │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  STAGE 2 · COMPARISON  (a.k.a. similarity scoring)      │    │
│  │  Compare your cleaned string to every cleaned candidate │    │
│  │  in the reference list. Each comparison returns a       │    │
│  │  number between 0 (nothing alike) and 1 (identical).    │    │
│  │                                                         │    │
│  │      "jhon smith"  vs  "john smith"   →  0.94           │    │
│  │      "jhon smith"  vs  "jon smyth"    →  0.84           │    │
│  │      "jhon smith"  vs  "wei chen"     →  0.18           │    │
│  │      ... 22 candidates in total                         │    │
│  └─────────────────────────────────────────────────────────┘    │
│      │                                                          │
│      ▼                                                          │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  STAGE 3 · RANKING  (a.k.a. picking the winner)         │    │
│  │  Sort candidates by score, highest first. The top one   │    │
│  │  is the likely match.                                   │    │
│  │                                                         │    │
│  │      N°1  John Smith   0.94   ◄── likely match          │    │
│  │      N°2  Jon Smyth    0.84                             │    │
│  │      N°3  ...                                           │    │
│  └─────────────────────────────────────────────────────────┘    │
│      │                                                          │
│      ▼                                                          │
│  THE LIKELY MATCH                                               │
│  "John Smith"                                                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

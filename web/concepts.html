What is fuzzy matching?

Fuzzy matching is what an analyst reaches for when an exact match isn't enough — when typos, abbreviations, or reordered words stop two obviously-related strings from comparing equal. If your day-to-day involves deduplicating records, cleaning up a customer list, or joining free-text data against a reference table, this page covers where fuzzy matching fits in that work, what you need lined up before you run it, and how it actually decides what counts as a match.

Fuzzy matching is how a computer figures out that two strings probably mean the same thing, even when they aren't identical. "Jhon Smith" almost certainly meant "John Smith". A human reads that instantly. A computer needs a small pipeline.

N°1 Before you start

The analyst's checklist

Six things you want lined up before you run the pipeline. Skipping any of them tends to show up later as a confusing result or a meeting about whether a match is "really" a match.

# What you need Why Example
1 Input strings The messy data you want to match. Can be one entry, a spreadsheet column, or a stream. Jhon Smith, 10 smith st
2 Reference list The canonical, authoritative values you're matching against. Without this, there's nothing to match to. A customer master list, postcode register, product catalogue
3 Knowledge of the noise What kinds of errors do you actually expect? Drives which algorithm earns its keep. Typos? Reordered words? Abbreviations? Phonetic variants?
4 Normalisation rules Cleanups applied before any scoring. Must be identical on both sides — input and reference. Lowercase, strip punctuation, remove accents, expand St→Street
5 Confidence thresholds Cutoffs for auto-match, human review, and unmatched. A business decision, not a technical one. Only needed when fuzzy match runs — exact match is unambiguous and doesn't use a threshold. ≥0.90 auto · 0.70–0.90 review · <0.70 unmatched
6 A review process Where flagged matches go, who looks at them, how fast, and where the final answer is recorded. Queue in a sheet → reviewed daily → outcome written back to source
7 A candidate-generation strategy Only matters if the reference list is large. You can't realistically score one input against a million records — you need a way to shortlist first. See the note below the workflow. Block by surname initial, postcode prefix, or trigram index
N°2 Where it sits & when it runs

The five-step workflow

Data cleansing has three branches: validation, standardisation, and record linkage. Fuzzy matching lives inside record linkage — but it only fires when exact matching has already failed. Here is the whole pipeline as a single picture.

Step Name What it does Example
1 Validate Is the value sensible at all? Reject obviously broken records before you do anything else. Email contains @; postcode is 4 digits
2 Standardise Force every record into the same format so later steps can compare like-for-like. JOHN SMITH → John Smith
3 Exact match Is the value in the reference list verbatim? Cheap, fast, unambiguous — try this first. John Smith == John Smith ✓
4 Fuzzy match this tool When exact fails, score similarity 0–1 against every candidate in the reference list. Jhon Smith → John Smith (0.94)
5 Apply threshold Decide what the score means: auto-match, send to human review, or leave unmatched. 0.94 → auto-match

Standardisation vs normalisation — they overlap in technique but live at different levels. Standardisation (Step 2 above) is the dataset-level practice: every phone number in our system should look the same. Normalisation is the per-string step inside Step 4, applied to a single value just before it gets compared. You'll see normalisation in the pipeline below.

In practice You don't score against millions

The diagram shows fuzzy match producing a score for every candidate in the reference list. That works for the 22 entries in this tool. It does not work for a CRM with a million customers — that would be a million similarity calculations per messy input.

In production, fuzzy match runs as a two-step process inside Step 4:

  1. Candidate generation (a.k.a. blocking) — cheaply shortlist a few dozen candidates that could be a match. Common techniques: blocking keys (only compare records sharing the same surname initial or postcode prefix), trigram indexes (PostgreSQL pg_trgm, Elasticsearch), or locality-sensitive hashing (MinHash / LSH).
  2. Scoring — run the expensive similarity algorithms on the shortlist only.

A good blocking strategy turns a million-row problem into a fifty-row problem before the similarity algorithm sees a single character. If you're working with a real-world reference list, look at libraries like recordlinkage, dedupe, or splink — they handle blocking and scoring as one pipeline.

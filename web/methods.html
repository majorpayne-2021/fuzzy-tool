The algorithms

Plain-English explanations of the four similarity-scoring algorithms used inside Stage 2 of the fuzzy-matching pipeline. No code, no math — just the intuition and when each one earns its keep.

Why more than one algorithm?

There's no single "best" similarity score. Each algorithm measures a different kind of similarity. A name with a typo needs one approach; a sentence with words reordered needs another; a partial match inside a longer string needs a third. The skill is picking the right tool — or combining several — for the noise you actually have.

The four algorithms below all sit inside Stage 2 of the pipeline on the concepts page. They assume the strings have already been normalised.

N°1 Method

Levenshtein

Edit distance

Count the fewest single-character edits — insertions, deletions, and substitutions — needed to turn one string into the other.

Analogy

Think of it as counting keystrokes. How many characters would you have to change to fix a typo? That's the distance. Divide by the length of the longer string to turn the count into a 0-to-1 similarity score.

Worked example
"kitten" → "sitting" requires 3 edits:
1. k → s (substitution)
2. e → i (substitution)
3. insert g at the end
Distance 3, length 7 → similarity ≈ 0.57.
When to use
  • Typos in short-to-medium strings (names, product codes).
  • Spelling variations of the same word in the same language.
  • When every character matters and position is meaningful.
When it fails
  • Word reordering ("John Smith" vs "Smith, John" scores near zero).
  • Very different string lengths — many edits even if the content is similar.
  • Phonetic differences ("Catherine" vs "Katherine" starts losing ground fast).
In your own project

Python: rapidfuzz (fast) or python-Levenshtein. JavaScript: fastest-levenshtein. Almost every language has a well-tested implementation — you shouldn't write your own unless you're learning.

N°2 Method

Jaro-Winkler

Edit distance (specialized)

Like Levenshtein, but purpose-built for short strings like names, and weighted to reward pairs that start with the same characters.

Analogy

People typo the ends of names more often than the beginnings. Jaro-Winkler bakes this into the score: strings that share a prefix get a bonus. "Jonathan" and "Jonathon" score higher than a character count alone would predict, because the first five letters match.

Worked example
"John" vs "Jon" — matching characters within a small window, 0 transpositions, shared prefix of 3 ("Jon"). Base Jaro ≈ 0.92, plus the prefix bonus → final ≈ 0.94.
When to use
  • Short strings where order matters: names, product SKUs, place names.
  • When you want to weight "same start" as evidence of a match.
  • Record-linkage tasks (deduplicating customer or patient records).
When it fails
  • Long strings like sentences — the prefix weighting stops helping.
  • Word reordering — same weakness as Levenshtein.
  • Phonetic variations — combine with a phonetic method if you need those.
In your own project

Python: jellyfish, rapidfuzz. JavaScript: natural, string-similarity-js.

N°3 Method

Jaccard similarity (tokens)

Set-based

Treat each string as a bag of words. Similarity = how many words they share, divided by the total number of unique words across both.

Analogy

Ignore word order entirely. "John Smith" and "Smith, John" contain the same words, so they're the same to this method. Great when human data entry scrambles the order but keeps the content.

Worked example
"red blue green" vs "green red yellow"
Shared: {red, green}. Total unique: {red, blue, green, yellow}.
Jaccard = 2 / 4 = 0.50.
When to use
  • Text where word order varies but the words themselves carry the meaning.
  • Short notes, tags, keyword lists, concatenated name fields.
  • As one input when combining multiple methods.
When it fails
  • Typos within words — "apple" and "appel" are totally different tokens.
  • Single-word strings — the set has one element, so it's all-or-nothing.
  • Different vocabularies that mean the same thing — "car" vs "automobile".
In your own project

A five-line function in any language. For large-scale near-duplicate detection, look up MinHash / LSH.

N°4 Method

Jaccard similarity (n-grams)

Set-based

Same idea as Jaccard tokens, but instead of whole words you use short overlapping character sequences. Catches similarity inside words.

Analogy

Slide a small window across the string and collect every length-n chunk. "Street" and "St" share the 2-gram "St" — enough to catch a partial match. This is what powers a lot of "typo-tolerant" search under the hood.

Worked example
"hello" bigrams: {he, el, ll, lo}
"hallo" bigrams: {ha, al, ll, lo}
Shared: {ll, lo}. Total unique: 6. Jaccard = 2 / 6 ≈ 0.33.
When to use
  • Fuzzy search and autocomplete.
  • Finding near-duplicate text blocks.
  • Detecting partial matches inside long strings.
When it fails
  • Very short strings — a 2-letter string has only one 2-gram.
  • Totally different vocabularies — shares no n-grams at all.
  • When position matters. Order of n-grams is lost.
In your own project

Easy to write by hand, or look for ngram utilities in your language's standard library. For big-scale search, inverted indexes over n-grams are a classic trick.

Combining algorithms in practice

Real pipelines often run several algorithms at once and either take the highest score, a weighted sum, or feed all the scores into a classifier. The tool shows every algorithm's score side-by-side for each candidate, so you can see how they agree and disagree on the same pair of strings.

For when each score crosses into "auto-match" or "needs human review", see the threshold cascade on the concepts page.
